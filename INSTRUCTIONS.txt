  PICTOOL v3 — Quran Screenshot Organizer
  Instructions for all users

WHAT CHANGED IN v3 (READ THIS FIRST)
────────────────────────────────────
v2 kept everyone's data in shared files inside a Google Drive
folder. That design could silently destroy work: if someone's
Drive had been offline and then came back, their stale files
could overwrite everyone else's newer data. This actually
happened.

v3 removes that risk by design:

  • DO NOT sync the images folder with Google Drive anymore.
    Each person keeps their OWN local copy of the folder.

  • Each person has exactly ONE data file that only their own
    machine ever writes:
        A → a_data.json
        Z → z_data.json
        S → s_data.json

  • Nothing writes to your file in the background. Ever.
    Only you, on your machine, through the app.

  • You see other people's data only when YOU choose to import
    their file. Nothing arrives behind your back.

The worst case is now "I'm looking at slightly old data from
the other person" — never "my work is gone."


SETUP (once per machine)
────────────────────────
1. Open the app: https://<your-github-user>.github.io/pictool/
   (Chrome or Microsoft Edge only.)

2. Choose who you are (A, Z, or S). This is stored on your
   machine and decides the single file the app writes to.
   Choose carefully — changing it later is deliberately
   protected, because writing another person's file is exactly
   the mistake v3 exists to prevent.

3. Click "فتح مجلد الصور" and select your local images folder.
   The app remembers it next time.

4. On first run, the app builds your personal data file from
   the old v2 files (tags.json / *_states.json) automatically.
   It never modifies those old files.


DAILY USE
─────────
• Open the app link, work normally: tag surahs, set status,
  write notes and units. Every change saves to YOUR file only.

• "عرض بيانات": toggle whether you want to see A's / Z's / S's
  data alongside yours. This only affects the display — it can
  never change what gets written.

• Next to those buttons the app shows how fresh each person's
  imported data is ("بيانات Z: 16/7 21:38"). It turns orange
  after 3 days and red after 7, so stale data is never silent.


CORRECTING A WRONG SURAH TAG
────────────────────────────
Surah tags are shared: a picture either contains a surah or it
doesn't, so EITHER A or Z can fix a wrong tag, and the fix
reaches the other person.

How it works: the app remembers each tag change you make and
when you made it. When files are exchanged, the most RECENT
change for each picture+surah wins. So:

  • Z removes a wrong surah on Tuesday. A imports Z's file →
    the tag disappears for A too, even though A's own file
    still had it.
  • If A disagrees and re-adds it on Wednesday, that's newer,
    so it comes back for both.

Hover any tag to see who last changed it and when — so if a
tag ever flips back, you can see who did it and talk about it.

Importing an OLD file can never resurrect a deleted tag or
undo a newer correction: old changes simply lose.

Status, notes and work units are different — those are
personal, and each person keeps their own. They are shown side
by side, never merged.


FLAGGING A PICTURE FOR THE OTHER PERSON
───────────────────────────────────────
To point the other person at pictures you both need to look at:

1. Once, create a shared status tag for it:
   "🏷 إدارة الوسوم" → add e.g. "للمناقشة" (needs discussion).
   Status tags are shared, so it appears for BOTH of you after
   the next file exchange — you only create it once.

2. Mark the pictures with that status, and write a note saying
   what you want (notes are personal, so yours is clearly
   yours: "Z: هذه تحتاج مراجعة معك").

3. Export and send your file. They import it.

4. They then filter: "الحالة: للمناقشة" plus your letter under
   "تصفية صور موسومة من:" — that lists exactly the pictures you
   flagged, with your note visible on each.

Note the difference: the status TAG (the vocabulary) is shared,
but WHO marked WHICH picture is personal. So both of you can
flag pictures independently and see each other's flags side by
side — nothing is overwritten, and you can always tell who
flagged what.


SHARING YOUR WORK WITH THE OTHERS
─────────────────────────────────
1. You: click "📤 إرسال بياناتي".
   Your system's share menu opens — pick WhatsApp, Mail,
   AirDrop, or anything else, and send it straight to the
   other person.
   If your browser can't share files, the file is downloaded
   instead and you attach it yourself. Either way the filename
   says who it's from and when (e.g. a_data_2026-07-17_21-40-05-123.json).

2. Them: click "📥 استيراد" and pick the file they received.

The app reads the owner's identity from INSIDE the file, not
from its name — so a renamed file still lands in the right
place, and importing someone's file can only ever update THAT
person's data. An import can never touch your own work.

If you import a file older than what you already have from
that person, the app warns you first.


BACKUPS AND VERSION HISTORY
───────────────────────────
• Every time you open your folder, the app saves a timestamped
  snapshot of your data file into the "backups" subfolder,
  before anything is written that session.

• Every file you import is archived there too.

• A snapshot is also taken every 12 hours while you work, so
  long sessions are covered too.

• Click "🕘 النسخ الاحتياطية" to see every version with a
  summary (owner, save date, how many surahs / statuses /
  notes / units it holds) and restore any of them.

• COMPARING VERSIONS: tick one version to compare it with your
  current data, or tick two to compare them with each other.
  You get a readable list of what actually differs — surahs
  added or removed, statuses and notes and units changed, work
  gained or lost. Use this before restoring, so you always know
  what you are about to change.

• Backups are NEVER deleted automatically. When you want to
  clear space, a button appears offering to delete versions
  older than 30 days — it shows you exactly what would go and
  asks first. The newest few are always kept, and files the
  app doesn't recognise are never touched.

• Restoring first backs up your current state, then swaps.
  So restoring can never lose anything either — including
  restoring by mistake.

The backups folder is small (kilobytes per version). Leave it
alone; don't delete it.


IMPORTANT RULES
───────────────
• Do NOT put the images folder back into Google Drive or any
  other sync tool. That reintroduces the exact problem v3
  solves.

• Do NOT copy someone else's *_data.json into your folder by
  hand. Use "📥 استيراد" so the app can validate, archive,
  and warn about it.

• Deleting images is disabled: everyone's folder copy must
  stay aligned by hand, so silent deletions are not allowed.

• User S is read-only and cannot write any data.


BROWSER REQUIREMENT
───────────────────
Chrome or Microsoft Edge only.
Does NOT work in Firefox or Safari.
